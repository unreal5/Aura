\part{Attribute Set}

\section{\texttt{PreAttributeChange}}

\textbf{核心结论：}
$\begin{cases}
	\text{CurrentValue}\Rightarrow\text{PreAttributeChange} \\
	\text{BaseValue}\Rightarrow\text{PreAttributeBaseChange}
\end{cases}$

\begin{myenum}
	\item \textbf{两者触发时机}

		\begin{itemize}
			\item \ci{PreAttributeChange}在属性的 \textbf{当前值（CurrentValue）} 即将变化时触发。  
			常见于带持续时间/无限时长 GE 的修饰器引起的数值变化（聚合后更新 Current）。
			
			\item \ci{PreAttributeBaseChange}
			在属性的 \textbf{基础值（BaseValue）} 即将变化时触发。  
			常见于 Instant GE 或直接改 Base 的路径（如 \texttt{SetNumericAttributeBase}）。
		\end{itemize}

\item \textbf{为什么会有两个钩子？}

在 GAS 里通常可理解为：\colorbox{yellow!5}{$
\text{CurrentValue} = \mathbf{f}(BaseValue, Modifiers)
$}
因此：
\begin{itemize}
	\item 改 \textbf{BaseValue} 前，用 \texttt{PreAttributeBaseChange}；
	\item 改 \textbf{CurrentValue} 前，用 \texttt{PreAttributeChange}。
\end{itemize}

\item \textbf{实战建议（很重要）}

\begin{itemize}
	\item 如果你要做夹逼（clamp），例如生命值限制在 \([0, MaxHealth]\)，
	最稳妥做法是 \textbf{两个函数都处理}，避免不同修改路径漏掉。
	\item 对 Instant GE 的“最终结果”校正，通常还会配合下面这个收尾钩子：
	\begin{itemize}
		\item \ci{PostGameplayEffectExecute}
	\end{itemize}
\end{itemize}
\end{myenum}

在 GAS 中，带 Period 的 GE 每次周期触发会按“执行(Execute)”语义处理，这条路径本质上接近 Instant GE，通常会动到 BaseValue。 而\textbf{一旦 BaseValue 变了，聚合器会重新计算 CurrentValue}，于是你通常会看到：
\begin{itemize}
	\item \ci{PreAttributeBaseChange(...)}：Base 将要改；
	\item \ci{PreAttributeChange(...)}：Current 随后重算将要改。
\end{itemize}
所以“两个都调用”是正常现象，不是异常。

可以把流程理解成：\colorbox{yellow}{$
\text{Periodic Tick} \Rightarrow \Delta \text{BaseValue} \Rightarrow \text{Recompute CurrentValue}$}。

因此：\ci{PreAttributeBaseChange} 先触发，\ci{PreAttributeChange} 后触发。

\paragraph{为什么你会觉得“按道理只改 Current”？}
因为“持续时间/无限时长 GE”这类描述通常让人联想到“临时 Buff/Debuff（Modifier 挂在 Active GE 上）”，那条路径主要表现为 Current 层变化。
但一旦你加了 Period，每个周期会走一次执行逻辑，常常落到 Base 改动路径上，所以就把两边都带起来了。

实战建议（Aura 项目里很实用）
\begin{enumerate}
	\item \ci{PreAttributeBaseChange}：放“基础值合法化”逻辑（如防 NaN、下限保护）。
	\item \ci{PreAttributeChange}：放“显示值 / 运行时值的 clamp”（如 Health 限制到 [0, MaxHealth]）。
	\item \ci{PostGameplayEffectExecute}：处理“执行后副作用”（死亡判定、受击、溢出转护盾等）。
	\item 避免在两个 \ci{Pre} 里写同一套重逻辑，否则容易重复处理。
\end{enumerate}

\section{PostGameplayEffectExecute}
\subsection{基础信息}
该函数使用参数\ci{struct FGameplayEffectModCallbackData}，定义如下：
\begin{code}
struct FGameplayEffectModCallbackData
{
	`$\dots$`
	const struct FGameplayEffectSpec&		EffectSpec;		// The spec that the mod came from
	struct FGameplayModifierEvaluatedData&	EvaluatedData;	// The 'flat'/computed data to be applied to the target
	
	class UAbilitySystemComponent &Target;		// Target we intend to apply to
};
\end{code}
可以从此结构中获取许多信息：哪个GE触发这个函数？效果应用在谁身上\sn{Target引用}？及最重要的\ci{EvaluatedData}，影响哪些数值。
\subsubsection{详解ATTRIBUTE\_ACCESSORS\_BASIC宏}
当使用宏：\ci{ATTRIBUTE_ACCESSORS_BASIC(ThisClass, Health)}时，会展开成如下代码（以下做了适当简化，避免干扰）：
\begin{code}[caption={简化的宏展开}]

static FGameplayAttribute GetHealthAttribute()
{
	static FProperty* Prop = FindFieldChecked<FProperty>(ThisClass::StaticClass(),
	FName("Health"));
	return Prop;
}

inline float GetHealth() const { return Health.GetCurrentValue(); }

inline void SetHealth(float NewVal)
{
	UAbilitySystemComponent* AbilityComp = GetOwningAbilitySystemComponent();
	AbilityComp->SetNumericAttributeBase(GetHealthAttribute(), NewVal);
}

inline void InitHealth(float NewVal)
{
	Health.SetBaseValue(NewVal);
	Health.SetCurrentValue(NewVal);
}
\end{code}
这个宏其实内联生成4个函数，分别对应属性及值的\ci{Init/Get/Set}函数。尤其要注意\ci{SetHealth}，它设置的是\textbf{Base值}。别忘记上节所述，Base值变动会引起Current值的变动\sn{Current值相当于$f(BaseValue,Modifiers)$}。
\begin{code}[caption={Base值变动，触发所有当前激活的GE重新设置Base值}]
void UAbilitySystemComponent::SetNumericAttributeBase(const FGameplayAttribute &Attribute, float NewFloatValue)
{
	// Go through our active gameplay effects container so that aggregation/mods are handled properly.
	ActiveGameplayEffects.SetAttributeBaseValue(Attribute, NewFloatValue);
}	
\end{code}



\subsection{1. 调用时机}

\ci{PostGameplayEffectExecute}
是在 \texttt{GameplayEffect} \textbf{已经执行完并且属性值已经被修改之后} 调用的回调。

可简单理解为：
	GameplayEffect 改值完成 $\Rightarrow$ \ci{PostGameplayEffectExecute}

它通常对应：
\begin{itemize}
	\item Instant GE
	\item Periodic GE 的每次周期结算
	\item ExecutionCalculation 最终写入属性之后
\end{itemize}
\begin{marker}
	 Duration GE 不触发 \ci{PostGameplayEffectExecute}
\end{marker}
\subsection{2. 适合做什么}

这个函数最适合做“\textbf{最终结算}”：

\begin{itemize}
	\item 对属性做 Clamp
	\item 处理伤害、治疗、蓝量结算
	\item 死亡判定
	\item 根据 \verb|Data| 读取来源、目标、标签等上下文
\end{itemize}

例如：
\begin{itemize}
	\item \verb|Health| 改变后限制到 \verb|[0, MaxHealth]|
	\item \verb|Mana| 改变后限制到 \verb|[0, MaxMana]|
\end{itemize}

\subsection{3. 与其他回调的区别}

\subsubsection{(1) 与 \texttt{PreAttributeChange} 的区别}

\begin{itemize}
	\item \verb|PreAttributeChange|：属性\textbf{将要}变化时调用
	\item \ci{PostGameplayEffectExecute}：属性\textbf{已经}变化后调用
\end{itemize}

所以：

\begin{itemize}
	\item \verb|PreAttributeChange| 更像“前置处理”
	\item \ci{PostGameplayEffectExecute} 更像“后置结算”
\end{itemize}

\subsubsection{(2) 与 \texttt{OnRep\_XXX} 的区别}

\begin{itemize}
	\item \ci{PostGameplayEffectExecute}：偏服务端结算逻辑
	\item \verb|OnRep_Health| / \verb|OnRep_Mana|：偏客户端同步与 UI 刷新
\end{itemize}

即：

\begin{quote}
	\ci{PostGameplayEffectExecute} 负责“规则结算”，  
	\verb|OnRep| 负责“复制后响应”。
\end{quote}

\subsection{4. 常见写法}

最常见的写法是根据本次被修改的属性分支处理：

\begin{code}
if (Data.EvaluatedData.Attribute == GetHealthAttribute())
{
	// 处理 Health
}
	
if (Data.EvaluatedData.Attribute == GetManaAttribute())
{
	// 处理 Mana
}
\end{code}

含义是：

\begin{quote}
	检查这次 GameplayEffect 实际改动的是哪个属性，然后分别做对应结算。
\end{quote}

\subsection{5. 注意事项}

\subsubsection{(1) 不是所有属性变化都会走这里}

\textbf{不要把它理解成“任意属性变化通知”。}

它主要是在 \verb|GameplayEffect| 执行并写入属性时触发。  
如果属性是通过别的方式修改的，不一定会调用这里。

\subsubsection{(2) 更适合做服务端权威逻辑}

常见放在这里的逻辑有：

\begin{itemize}
	\item Clamp
	\item 扣血/回血
	\item 死亡判定
	\item 资源修正
\end{itemize}

不建议把大量 UI 或表现层逻辑直接堆在这里。

\subsubsection{(3) 小心重复结算}

如果你在这里又施加新的 GE，或者再次修改相关属性，要注意：

\begin{itemize}
	\item 是否会重复触发
	\item 是否会形成递归
	\item 是否会导致一次伤害被算两遍
\end{itemize}

\subsubsection{(4) 推荐用于最终兜底}

对于生命值、蓝量这类有上下限的属性，  
\ci{PostGameplayEffectExecute} 很适合做最终 Clamp，保证结果合法。

\subsection{6. 一句话总结}

\begin{quote}
	\textbf{\ci{PostGameplayEffectExecute} 就是 GameplayEffect 改完属性后的最终结算点。}
\end{quote}

它最适合做：

\begin{itemize}
	\item Clamp
	\item 伤害/治疗收尾
	\item 死亡与资源判定
\end{itemize}

使用时要注意：

\begin{itemize}
	\item 它不是所有属性变化的统一入口
	\item 它更偏向服务端数值逻辑
	\item 客户端 UI 更新通常应结合 \verb|OnRep|
\end{itemize}

\subsection{对 PostGameplayEffectExecute 的准确理解}

可以近似理解为：

\begin{quote}
	\ci{PostGameplayEffectExecute} 是 \textbf{GameplayEffect 执行型结算之后} 的回调。
\end{quote}

它的触发关键并不是“属性改了 \verb|BaseValue|”本身，而是：

\begin{center}
	\textbf{该次属性修改是否来自 GameplayEffect 的 Execute 路径}
\end{center}

在 GAS 中，典型会触发它的情况有：

\begin{itemize}
	\item Instant GameplayEffect
	\item Periodic GameplayEffect 的每次周期执行
	\item ExecutionCalculation 的最终属性写入
\end{itemize}

这些情况通常都会修改属性的 \verb|BaseValue|，因此可以得到一个常见现象：

\begin{quote}
	\textbf{凡是触发 PostGameplayEffectExecute 的，大多数时候都伴随着 \ci{BaseValue} 的修改。}
\end{quote}

但反过来不能机械理解为：

\begin{quote}
	“只要 \verb|BaseValue| 变了，就一定会触发 \ci{PostGameplayEffectExecute}。”
\end{quote}

更不能把它当成“所有属性变化的统一通知入口”。

特别是：

\begin{itemize}
	\item \verb|Duration| / \verb|Infinite| 类型的 GameplayEffect 通常只是通过 Modifier 影响 \verb|CurrentValue|
	\item 这类变化一般不会调用 \ci{PostGameplayEffectExecute}
\end{itemize}

因此，一句话总结为：

\begin{quote}
	\textbf{PostGameplayEffectExecute 本质上是 Execute 型 GameplayEffect 的结算后回调，而不是单纯的 BaseValue 变化回调。}
\end{quote}

\subsection{什么是 GameplayEffect 的 Execute 路径}

所谓 \textbf{Execute 路径}，可以理解为：

\begin{quote}
	\textbf{GameplayEffect 不是仅仅挂上一个持续 Modifier，而是实际执行了一次数值结算，并将结果真正写入属性。}
\end{quote}

其核心过程可概括为：

\begin{center}
	计算修改量 $\rightarrow$ 执行属性写入 $\rightarrow$ 触发结算后回调
\end{center}

在 GAS 中，\ci{PostGameplayEffectExecute} 主要就是在这种“\textbf{执行型结算完成之后}”被调用的。

\subsubsection{1. Execute 路径的直观理解}

可以把 Execute 路径理解成“\textbf{结算落账}”：

\begin{itemize}
	\item 火球命中，立刻扣除 30 点生命
	\item 治疗命中，立刻恢复 20 点生命
	\item 周期伤害每跳一次，真正扣除一次生命
\end{itemize}

这类变化的特点是：

\begin{itemize}
	\item 它是一次真实的数值事件
	\item 结算结果会沉淀到属性中
	\item 通常对应属性 \verb|BaseValue| 语义上的变化
	\item 通常会触发 \ci{PostGameplayEffectExecute}
\end{itemize}

\subsubsection{2. 与 Modifier 路径的区别}

与 Execute 路径相对的是“\textbf{持续 Modifier 生效路径}”，它更像“\textbf{状态挂账}”：

\begin{itemize}
	\item 角色身上挂了一个 Buff，于是当前生命临时 +50
	\item 身上挂了一个 Debuff，于是当前护甲临时 -10
	\item 光环存在期间，移速持续乘以 1.2
\end{itemize}

这类变化更像：

\begin{quote}
	效果存在时，属性的最终生效值被持续影响；效果消失时，影响随之消失。
\end{quote}

因此它通常：

\begin{itemize}
	\item 影响的是 \verb|CurrentValue|
	\item 不属于一次“执行结算写入”
	\item 一般不会调用 \ci{PostGameplayEffectExecute}
\end{itemize}

\subsection{为什么说它“通常对应 BaseValue”，但又不能简单等同}

很多时候可以观察到：\colorbox{yellow}{
	$触发  \texttt{PostGameplayEffectExecute} \Rightarrow 常常伴随 \text{BaseValue} 变化$}

因此容易形成一种简化理解：

\begin{quote}
	“只有修改 \verb|BaseValue| 的才会触发 \ci{PostGameplayEffectExecute}。”
\end{quote}

这个理解\textbf{接近正确}，但更准确的说法应当是：

\begin{quote}
	\textbf{\ci{PostGameplayEffectExecute} 的触发关键不是“BaseValue 变了”，而是“GameplayEffect 走了 Execute 路径”。}
\end{quote}

也就是说：

\begin{center}
	\textbf{Execute} $\Rightarrow$ \ci{PostGameplayEffectExecute}
\end{center}

而不是机械地写成：

\begin{center}
	\verb|BaseValue| \text{变化} $\Rightarrow$ \ci{PostGameplayEffectExecute}
\end{center}

\subsection{为什么 Duration 型 GE 不适合直接处理伤害结算}

如果一个 \commandbox{Duration GE} 只是给 \verb|Health| 挂了一个持续的负 Modifier，例如：

\[
CurrentHealth = BaseHealth + (-50)
\]

那么会出现两个问题：

\begin{enumerate}
	\item 它通常不会触发 \ci{PostGameplayEffectExecute}
	\item 效果消失时，生命值可能“弹回去”
\end{enumerate}

这会让“死亡判定”变得很不稳定，例如：

\begin{itemize}
	\item Debuff 生效时，目标当前生命低于 0
	\item Debuff 消失后，又恢复到正数
\end{itemize}

因此在战斗系统里，更常见也更稳妥的做法是：

\begin{itemize}
	\item \textbf{伤害}：使用 Instant 或 Periodic Execute 进行真实扣血
	\item \textbf{持续伤害}：使用 Periodic，每次 tick 都执行一次真实扣血
	\item \textbf{Buff / Debuff}：使用 Duration / Infinite 影响攻击、防御、速度、护盾等
\end{itemize}

\subsection{与 \texttt{PostGameplayEffectExecute} 的关系}

\ci{PostGameplayEffectExecute} 可以理解为：

\begin{quote}
	\textbf{Execute 型 GameplayEffect 完成属性结算后，AttributeSet 拿到的一次后置处理机会。}
\end{quote}

因此它特别适合做：

\begin{itemize}
	\item \verb|Health| Clamp
	\item \verb|Mana| Clamp
	\item 死亡判定
	\item 护盾扣减后的血量修正
	\item 伤害、治疗、资源变化的最终收尾
\end{itemize}

但不应把它理解成：

\begin{itemize}
	\item 所有属性变化的统一通知
	\item 所有 \verb|CurrentValue| 变化都会触发的回调
	\item 客户端 UI 更新的唯一入口
\end{itemize}

\subsection{结合 AuraAttributeSet 的理解}

在当前的 `UAuraAttributeSet` 中，\ci{PostGameplayEffectExecute} 被用于：

\begin{itemize}
	\item 当 \verb|Health| 被 Execute 型效果修改后，限制在 \verb|[0, MaxHealth]|
	\item 当 \verb|Mana| 被 Execute 型效果修改后，限制在 \verb|[0, MaxMana]|
\end{itemize}

这说明当前代码的设计思路是：

\begin{quote}
	\textbf{把 Health 和 Mana 当作需要“真实结算落地”的属性，并在结算后统一做兜底修正。}
\end{quote}

如果后续伤害系统继续沿着这个思路扩展，那么推荐做法是：

\begin{itemize}
	\item 用 Instant / Periodic Execute 处理扣血与回血
	\item 在 \ci{PostGameplayEffectExecute} 中处理 Clamp、死亡、受击收尾
	\item 不要用 Duration 直接给 \verb|Health| 挂持续负 Modifier 来表示真实伤害
\end{itemize}

\subsection{一句话总结}

\begin{quote}
	\textbf{Execute 路径 = GameplayEffect 真正执行了一次数值结算并把结果写入属性；而 \ci{PostGameplayEffectExecute} 就是这次结算之后的后置回调。}
\end{quote}

\section{Attribute Based Modifier 的基本用法与作用}

本节介绍了 Gameplay Effect 中另一种重要的数值计算方式：\texttt{Attribute Based}。此前课程中的 Modifier Magnitude 一直使用 \texttt{Scalable Float}，而 \texttt{Attribute Based} 的核心价值在于，它可以让一个属性的变化量直接基于其他属性的当前值来计算，因此非常适合实现“派生属性”或“由其他属性驱动的属性变化”。

为了演示这一点，课程先新建了一个测试用的 \texttt{Effect Actor}，并为其添加一个可见的 Box Collision，在角色进入碰撞体时调用 \texttt{ApplyEffectToTarget}。随后又创建了一个测试用的 Gameplay Effect，命名为 \texttt{GE\_Test\_Attribute\_Based}，并将其设置为该测试 Actor 的 Instant Gameplay Effect。这样，Aura 角色一旦进入该区域，就会立即受到这个效果的影响。

接着，课程重点讲解了如何配置 \texttt{Attribute Based} Modifier。示例中给 Health 添加一个 Modifier，操作类型保持为 \texttt{Add}，但将 Magnitude Calculation Type 从 \texttt{Scalable Float} 改为 \texttt{Attribute Based}。在其展开配置中，\texttt{Backing Attribute} 用来指定“以哪个属性为基础”进行计算，\texttt{Attribute Source} 则决定这个属性是从 Source 还是 Target 身上获取。由于测试场景中是将效果施加给 Aura，所以这里选择从 \texttt{Target} 捕获属性，并以 Aura 的 \texttt{Vigor} 作为基础属性。这样，当角色进入测试区域时，Health 就会增加当前 Vigor 的数值；示例中 Aura 的 Health 为 50、Vigor 为 9，因此进入后 Health 变为 59。课程还简要提到 \texttt{Snapshot} 用于决定何时捕获属性值，但这一机制会留到后面更合适的场景再深入讨论。

之后，课程进一步演示了多个 Modifier 同时作用于同一属性的情况。除了第一个基于 Vigor 的 Health 加成外，又添加了第二个同样作用于 Health 的 Modifier，这次改为基于 \texttt{Strength}。由于两个 Modifier 的操作类型都是 \texttt{Add}，最终效果就是把 Vigor 和 Strength 的值都加到 Health 上。示例中角色初始 Health 为 50，Vigor 为 9，Strength 为 10，因此进入测试区域后 Health 变为 69，结果符合直觉。

本节的结论是：\texttt{Attribute Based} 提供了一种非常灵活的方式，使 Gameplay Effect 的数值不再是固定常数，而是可以由角色自身或来源对象的属性动态驱动。当前示例展示的是最简单、最直观的加法情况，而更复杂的行为还涉及多个 Modifier 叠加时的执行顺序，以及 \texttt{Coefficient}、\texttt{Pre-Multiply Additive}、\texttt{Post-Multiply Additive} 等附加参数。下一节将继续深入分析不同 Modifier Operation 下的运算顺序与具体规则。

\section{SynchronizeProperties 的作用}

在 UMG / \texttt{UUserWidget} 体系中，\texttt{SynchronizeProperties()} 的核心职责可以概括为：

\begin{quote}
	把对象上保存的属性值，同步到实际的 Slate / Widget 显示层。
\end{quote}

它更偏向于“属性同步”而不是“业务逻辑初始化”。

很多时候可以把它理解为：

\begin{itemize}
	\item C++ 成员变量里已经有了一些值；
	\item 现在需要把这些值真正应用到界面控件上；
	\item 这一步就是 \texttt{SynchronizeProperties()} 要做的事情。
\end{itemize}

例如：

\begin{itemize}
	\item 把一个 \texttt{FText TitleText} 同步到 \texttt{TextBlock->SetText(...)}；
	\item 把一个 \texttt{float BoxWidth} 同步到 \texttt{SizeBox->SetWidthOverride(...)}；
	\item 把一个颜色变量同步到 \texttt{Border} 或 \texttt{Image} 上。
\end{itemize}

\section{它和 NativePreConstruct / NativeConstruct 的区别}

这三个函数都和 Widget 初始化有关，但关注点不同。

\subsection{NativePreConstruct}

\texttt{NativePreConstruct()} 更偏向：

\begin{itemize}
	\item 在设计器预览和运行时都可能调用；
	\item 用来提前调整 UI 的外观；
	\item 适合做“展示层的预配置”。
\end{itemize}

例如：

\begin{itemize}
	\item 设置默认文本；
	\item 设置尺寸；
	\item 设置可见性；
	\item 根据可编辑属性刷新外观。
\end{itemize}

\subsection{NativeConstruct}

\texttt{NativeConstruct()} 更偏向：

\begin{itemize}
	\item 运行时逻辑初始化；
	\item 绑定按钮事件；
	\item 获取游戏对象；
	\item 注册委托；
	\item 创建动态子控件。
\end{itemize}

它通常不是专门为“属性同步”设计的，而是为“控件开始参与游戏逻辑”准备的。

\subsection{SynchronizeProperties}

\texttt{SynchronizeProperties()} 则更聚焦于：

\begin{itemize}
	\item 将当前对象的属性值应用到实际控件；
	\item 保证界面显示和对象状态一致；
	\item 更像“刷新显示层”。
\end{itemize}

因此可以把它和前两个函数粗略区分成：

\begin{itemize}
	\item \texttt{NativePreConstruct()}：预构造阶段调整外观；
	\item \texttt{NativeConstruct()}：运行时接入逻辑；
	\item \texttt{SynchronizeProperties()}：把属性值同步到界面。
\end{itemize}

\section{什么时候会调用}

从使用者角度，不必死记底层所有细节，但需要知道一个重要事实：

\begin{quote}
	\texttt{SynchronizeProperties()} 不是“业务初始化入口”，而是“属性刷新入口”。
\end{quote}

它通常会在引擎认为“需要把对象属性重新应用到界面”时被调用。  
这类场景一般包括：

\begin{itemize}
	\item Widget 构建或重建时；
	\item 设计器预览需要刷新时；
	\item 某些属性变化后需要重新同步时。
\end{itemize}

所以它可能不是只调用一次，而可能被多次调用。

这也是为什么在这个函数里通常应该做：

\begin{itemize}
	\item 幂等的 UI 同步；
	\item 轻量的显示刷新；
	\item 不依赖复杂运行时上下文的操作。
\end{itemize}

\section{适合放什么代码}

\subsection{适合放的内容}

适合放进 \texttt{SynchronizeProperties()} 的，一般是“把已有属性值写到控件上”的代码，例如：

\begin{itemize}
	\item 文本同步；
	\item 颜色同步；
	\item 图片同步；
	\item 尺寸同步；
	\item 可见性同步；
	\item 某些简单状态的表现层刷新。
\end{itemize}

例如：

\begin{code}
	if (TitleTextBlock)
	{
		TitleTextBlock->SetText(TitleText);
	}
	
	if (SizeBox_Root)
	{
		SizeBox_Root->SetWidthOverride(BoxWidth);
		SizeBox_Root->SetHeightOverride(BoxHeight);
	}
\end{code}

这里的思想是：

\begin{quote}
	成员变量是“数据”，Widget 子控件是“显示”，\\
	\texttt{SynchronizeProperties()} 负责把前者写到后者。
\end{quote}

\subsection{不适合放的内容}

不建议放进 \texttt{SynchronizeProperties()} 的内容包括：

\begin{itemize}
	\item 获取 \texttt{PlayerController} / \texttt{GameMode} / \texttt{PlayerState}；
	\item 绑定按钮点击事件；
	\item 注册委托；
	\item 创建大量动态子控件；
	\item 依赖运行时世界状态的复杂逻辑；
	\item 带副作用的业务操作。
\end{itemize}

原因是：

\begin{enumerate}
	\item 这个函数可能被多次调用；
	\item 它的职责本来就是“同步属性”，不是“做业务初始化”；
	\item 如果在这里绑定事件或重复创建对象，很容易导致重复执行。
\end{enumerate}

\section{一个典型使用模式}

一个常见做法是：

\begin{enumerate}
	\item 在 \ci{UPROPERTY(EditAnywhere)} 里暴露一些可编辑参数；
	\item 在 \ci{SynchronizeProperties()} 中把这些参数应用到子控件；
	\item 这样在设计器和运行时都更容易保持显示一致。
\end{enumerate}

例如一个 Widget 有如下属性：

\begin{code}
	UPROPERTY(EditAnywhere, BlueprintReadWrite, Category="UI")
	float BoxWidth = 805.f;
	
	UPROPERTY(EditAnywhere, BlueprintReadWrite, Category="UI")
	float BoxHeight = 906.f;
\end{code}

那么就可以在 \texttt{SynchronizeProperties()} 中写：

\begin{code}
	void UAttributeMenu::SynchronizeProperties()
	{
		Super::SynchronizeProperties();
		
		if (SizeBox_Root)
		{
			SizeBox_Root->SetWidthOverride(BoxWidth);
			SizeBox_Root->SetHeightOverride(BoxHeight);
		}
	}
\end{code}

这样它的语义非常明确：

\begin{quote}
	每当需要同步属性时，就把 \texttt{BoxWidth} 和 \texttt{BoxHeight} 应用到 \texttt{SizeBox\_Root}。
\end{quote}

\section{和你的 AttributeMenu 场景如何对应}

从你当前的 \texttt{`AttributeMenu.h`} 来看，你已经有：

\begin{itemize}
	\item \texttt{SizeBox\_Root}
	\item \texttt{BoxWidth}
	\item \texttt{BoxHeight}
	\item \texttt{UpdateFrameSize()}
\end{itemize}

这类“根据可编辑属性同步 UI 尺寸”的行为，其实就很适合使用 \texttt{SynchronizeProperties()}。

可以把思路理解为：

\begin{itemize}
	\item \texttt{BoxWidth} / \texttt{BoxHeight} 是数据；
	\item \texttt{SizeBox\_Root} 是显示对象；
	\item \texttt{SynchronizeProperties()} 就是把这两个数据写入显示对象。
\end{itemize}

因此如果 \texttt{UpdateFrameSize()} 的职责仅仅是：

\begin{itemize}
	\item 调用 \texttt{SetWidthOverride()}；
	\item 调用 \texttt{SetHeightOverride()}；
\end{itemize}

那么它就非常适合在 \texttt{SynchronizeProperties()} 中调用。

例如逻辑上可以写成：

\begin{code}
	void UAttributeMenu::SynchronizeProperties()
	{
		Super::SynchronizeProperties();
		UpdateFrameSize();
	}
\end{code}

\section{为什么它在做“编辑器友好型 UI”时很有用}

当一个 Widget 需要在编辑器里就能看到配置效果时，\texttt{SynchronizeProperties()} 特别有价值。

因为这种场景里，你经常会有这样的需求：

\begin{itemize}
	\item 改一个 \texttt{UPROPERTY(EditAnywhere)}；
	\item 立即让对应控件的显示变化反映出来；
	\item 保持设计器预览和运行时显示尽量一致。
\end{itemize}

这正是“属性同步函数”最擅长做的事情。

所以对于：

\begin{itemize}
	\item 面板尺寸；
	\item 默认标题；
	\item 颜色主题；
	\item 图标样式；
	\item 文本样式；
\end{itemize}

这样的可编辑展示参数，\texttt{SynchronizeProperties()} 通常是很合适的切入点。

\section{使用时的注意事项}

\subsection{要调用父类实现}

通常应当先调用：

\begin{code}
	Super::SynchronizeProperties();
\end{code}

这样可以保证父类已有的同步逻辑正常执行。

\subsection{尽量保持幂等}

因为它可能被多次调用，所以最好保证：

\begin{itemize}
	\item 调一次和调多次结果一致；
	\item 不产生额外副作用；
	\item 不重复绑定事件；
	\item 不重复创建对象。
\end{itemize}

\subsection{注意空指针判断}

由于子控件可能在某些阶段尚未准备好，通常要先判断：

\begin{code}
	if (SizeBox_Root)
	{
		...
	}
\end{code}

\subsection{不要把它当成业务入口}

如果你需要：

\begin{itemize}
	\item 和角色属性系统通信；
	\item 绑定按钮事件；
	\item 初始化运行时数据源；
	\item 构建滚动列表内容；
\end{itemize}

这些通常更适合放到 \texttt{NativeConstruct()} 或专门的初始化函数中，而不是 \texttt{SynchronizeProperties()}。

\section{一个简明判断标准}

可以用下面这个原则区分：

\begin{quote}
	如果你的代码是在说“把这个属性显示到界面上”，就优先考虑 \texttt{SynchronizeProperties()}；\\
	如果你的代码是在说“让这个界面开始参与游戏逻辑”，就优先考虑 \texttt{NativeConstruct()}。
\end{quote}

\section{一句话总结}

\texttt{SynchronizeProperties()} 的本质是：

\begin{quote}
	\textbf{把 Widget 对象上的配置属性，同步到实际界面表现。}
\end{quote}

它适合做轻量、可重复、偏显示层的刷新；  
而不适合承担复杂的运行时初始化和业务逻辑。

\section{Gameplay Abilities 设置}
\subsection{Tag}
Gameplay Ability 是派生自 \texttt{UGameplayAbility} 的类，用来定义角色可执行的技能、法术或动作，以及这些能力在什么条件下可以被使用。与普通函数式动作不同，它是一个可实例化、可异步执行的对象，能够跨时间执行多阶段逻辑，并根据玩家或环境条件触发不同分支。

课程还强调了 Gameplay Ability 的几个核心特性：它原生支持网络复制与预测，内置 \textbf{消耗} 与 \textbf{冷却} 机制，例如课程中会用法力值作为施法消耗；它还能配合 \texttt{UAbilityTask} 执行异步任务，并通过委托驱动 Blueprint 中的多输出执行流，从而兼顾 C++ 性能与 Blueprint 的可视化设计灵活性。

最后，课程说明了能力的使用流程：能力需要先授予给 \texttt{AbilitySystemComponent}，此时会生成并同步一个 \texttt{Ability Spec}，其中包含能力类、等级和可在运行时变化的信息。能力被激活后会进入活动状态，直到自行结束或被外部取消。整体上，Gameplay Ability 是 GAS 的核心执行单元，后续课程将开始实现第一个实际能力。

\subsection{EGameplayAbilityNetExecutionPolicy 各标识符详解}

\ci{NetExecutionPolicy}类型为\ci{EGameplayAbilityNetExecutionPolicy} 是 GAS\sidenote{Gameplay Ability System}中用于描述“一个 Ability 在网络环境下由谁执行、何时在本地执行”的策略枚举。  
它关注的是\textbf{执行位置与时机}，而不是授权安全（那是 \ci{EGameplayAbilityNetSecurityPolicy} 的职责）。

在源码中，这个命名空间下定义了四个枚举值（\ci{Type}）：
\begin{itemize}
	\item \ci{LocalPredicted}
	\item \ci{LocalOnly}
	\item \ci{ServerInitiated}
	\item \ci{ServerOnly}
\end{itemize}

	\hline
	\textbf{标识符} & \textbf{直观含义} & \textbf{典型行为与适用场景} \\
	\hline
	\endfirsthead
	\hline

	\ci{LocalPredicted} &
	客户端可预测执行（本地先跑） &

	优点是手感好、延迟感低；代价是要处理预测回滚/修正。常用于高频、对操作反馈敏感的主动技能。 \\

	\textbf{不适合}改变权威游戏状态（伤害、Buff、生死判定等）。 \\
	
	\hline
	
	\ci{ServerInitiated} &
	服务器发起，再在本地端也执行 &
	由服务器作为发起者决定执行，随后在具备本地控制的一侧也可运行对应逻辑。  
	适合“必须由服务器先判定是否发生，但客户端仍需执行表现/后续流程”的能力，例如服务器决定触发后，客户端再播放对应动画与表现。 \\
	
	\hline
	
	\ci{ServerOnly} &
	仅服务器执行 &
	Ability 只在服务器运行，客户端不做本地执行。  
	权威性最强、实现最稳妥，适合强规则或高风险逻辑（最终判定、关键状态变更、反作弊敏感流程）；但客户端即时反馈较弱，通常需额外做表现补偿。 \\
	
	\hline

\textbf{逐项深入解释：}

\textbf{1) \ci{LocalPredicted}}\\
这是 GAS 中最常见、也最“网络游戏手感友好”的策略。核心思想是“\textbf{先响应，后校验}”：
\begin{enumerate}
	\item 客户端收到输入后立即进入能力执行路径（预测执行）；
	\item 同步向服务器发送激活请求；
	\item 服务器做权威判定并广播/复制结果；
	\item 若客户端预测与服务器权威结果不一致，则发生修正（包括状态纠正、表现回滚或重放）。
\end{enumerate}
它通常依赖\textbf{预测键}（Prediction Key）等机制来关联客户端预测与服务器确认。  
适用关键词：\textbf{低延迟手感、可预测、可回滚}。

\textbf{2) \ci{LocalOnly}}\\
该策略强调“\textbf{仅本地语义}”。它不以网络权威同步为目标，因此一般不应该承载会影响他人或全局一致性的状态变更。  
实践中常把它用于：
\begin{itemize}
	\item 本地 HUD/准星/提示逻辑；
	\item 本地输入态机切换；
	\item 纯客户端可视化流程（不决定真实战斗结果）。
\end{itemize}
如果把真实战斗判定写进 \ci{LocalOnly}，多人游戏中会出现状态不一致或被服务器覆盖的问题。

\textbf{3) \ci{ServerInitiated}}\\
与 \ci{LocalPredicted} 的“客户端先动”不同，\ci{ServerInitiated} 是“\textbf{服务器先定，再让本地参与执行}”。  
典型流程是：
\begin{enumerate}
	\item 服务器触发能力（可能由规则、AI、服务器事件驱动）；
	\item 对应客户端接收到能力相关执行/表现；
	\item 本地端运行需要本地上下文的部分（动画、镜头、音效等）。
\end{enumerate}
它兼顾权威性与本地表现，但交互即时性通常不如 \ci{LocalPredicted}。

\textbf{4) \ci{ServerOnly}}\\
该策略最“纯权威”：只有服务器真的执行能力逻辑。  
优势：
\begin{itemize}
	\item 一致性最好；
	\item 调试和安全边界清晰；
	\item 作弊面更小。
\end{itemize}
代价：
\begin{itemize}
	\item 客户端输入后需要等待网络往返才能看到结果；
	\item 若无额外视觉预反馈，手感偏“重”。
\end{itemize}
因此常配合独立的客户端表现层（非能力权威逻辑）来减轻延迟体感。

\textbf{与其他“Net Policy”概念的边界：}
\begin{itemize}
	\item \ci{NetExecutionPolicy}：\textbf{在哪执行、谁先执行}；
	\item \ci{NetSecurityPolicy}：\textbf{客户端是否被允许请求执行/终止}；
	\item \ci{ReplicationPolicy}：\textbf{能力实例状态是否复制给谁}。
\end{itemize}
三者共同决定一个 Ability 的网络行为，语义互补而非替代。

\textbf{选型建议（经验法则）：}
\begin{itemize}
	\item 强调操作即时反馈：优先考虑 \ci{LocalPredicted}；
	\item 纯本地功能：\ci{LocalOnly}；
	\item 必须服务器先判定再驱动表现：\ci{ServerInitiated}；
	\item 高权威/高安全/关键判定：\ci{ServerOnly}。
\end{itemize}

\textbf{一句话总结：}\\
\ci{EGameplayAbilityNetExecutionPolicy} 的四个标识符，本质是在“\textbf{手感（客户端先响应）}”与“\textbf{权威（服务器先/仅服务器）}”之间做不同取舍。

\section{EGameplayAbilityNetSecurityPolicy 各标识符详解}

\ci{EGameplayAbilityNetSecurityPolicy} 用来定义 GAS 中 Ability 的\textbf{网络安全边界}：  
客户端是否被允许请求“执行（Execution）”或“终止（Termination）”该 Ability。  

它回答的问题是：\textbf{客户端的请求，服务器要不要接受？}  
注意这与 \texttt{EGameplayAbilityNetExecutionPolicy} 不同：后者关注“在哪执行”；本策略关注“谁有权限请求执行/结束”。

在源码中，该命名空间下 \ci{Type} 枚举包含四个值：
\begin{itemize}
	\item \ci{ClientOrServer}
	\item \ci{ServerOnlyExecution}
	\item \ci{ServerOnlyTermination}
	\item \ci{ServerOnly}
\end{itemize}

\begin{longtable}{p{0.22\linewidth} p{0.26\linewidth} p{0.46\linewidth}}
	\hline
	\textbf{标识符} & \textbf{权限语义} & \textbf{具体影响（服务器如何处理客户端请求）} \\
	\hline
	\endfirsthead
	\hline
	\textbf{标识符} & \textbf{权限语义} & \textbf{具体影响（服务器如何处理客户端请求）} \\
	\hline
	\endhead
	
	\ci{ClientOrServer} &
	客户端与服务器都可请求执行与终止 &
	客户端发起“激活请求”可被服务器接受；客户端发起“取消/结束请求”也可被服务器接受。  
	限制最少，开发最灵活，但安全约束最弱。 \\
	
	\hline
	
	\ci{ServerOnlyExecution} &
	执行权仅服务器可决定；终止仍可由客户端请求 &
	客户端请求“执行”会被服务器忽略；但客户端请求“取消/终止”仍可被服务器接受。  
	适合不希望客户端随意开技能，但允许客户端主动中断（例如松键取消、蓄力中止）。 \\
	
	\hline
	
	\ci{ServerOnlyTermination} &
	终止权仅服务器可决定；执行仍可由客户端请求 &
	客户端请求“执行”可被服务器接受；客户端请求“取消/结束”会被服务器忽略。  
	适合允许客户端触发，但结束时机必须由服务器掌控（例如必须完成服务器判定流程）。 \\
	
	\hline
	
	\ci{ServerOnly} &
	执行与终止都仅服务器可决定 &
	客户端对该 Ability 的执行/终止请求均被服务器忽略。  
	安全最强、权威最清晰，适合关键玩法与反作弊敏感逻辑。 \\
	
	\hline
\end{longtable}

\textbf{逐项深入理解：}

\textbf{1) \ci{ClientOrServer}}\\
这是最宽松策略。客户端对 Ability 的“开始”和“结束”都有请求权。  
优点是交互灵活、实现成本低；缺点是服务器对客户端行为限制最弱，不适合高价值判定（例如关键伤害结算、排名相关核心技能）。

\textbf{2) \ci{ServerOnlyExecution}}\\
本质是“\textbf{开关分离}”中的“开”受限：  
\begin{itemize}
	\item 客户端不能决定“开技能”；
	\item 客户端可以请求“关技能”。
\end{itemize}
典型用途是防止客户端伪造激活条件（距离、资源、冷却），但仍保留本地操作体验（玩家可以快速取消当前动作）。

\textbf{3) \ci{ServerOnlyTermination}}\\
与上一个相反，是“\textbf{关}受限”：  
\begin{itemize}
	\item 客户端可以请求激活；
	\item 客户端不能决定结束，服务器决定何时结束。
\end{itemize}
适合“启动可交给客户端，但结束必须满足服务端约束”的能力，比如通道技能、多段流程、必须完成服务端检查后才可终止的技能状态。

\textbf{4) \ci{ServerOnly}}\\
最严格策略：客户端既不能请求执行，也不能请求终止。  
服务器完全掌控该 Ability 生命周期。  
它通常用于：
\begin{itemize}
	\item 强对抗与反作弊敏感能力；
	\item 核心状态机切换（生死、变身、强控制等）；
	\item 需要绝对一致性的关键逻辑。
\end{itemize}

\textbf{与 \ci{NetExecutionPolicy} 的关系（高频混淆点）：}
\begin{itemize}
	\item \ci{NetExecutionPolicy}：定义能力\textbf{在哪边执行、是否预测执行}；
	\item \ci{NetSecurityPolicy}：定义客户端\textbf{是否有权请求执行/终止}；
	\item 两者叠加后，才是完整网络行为。
\end{itemize}
因此同一个 “看起来在客户端触发” 的 Ability，是否真的能被服务器接受，最终还要看 \ci{NetSecurityPolicy}。

\textbf{实战选型建议：}
\begin{itemize}
	\item 原型期/低风险能力：可先用 \ci{ClientOrServer} 提高迭代效率；
	\item 要防止“非法激活”：选 \ci{ServerOnlyExecution}；
	\item 要防止“非法中断”：选 \ci{ServerOnlyTermination}；
	\item 核心竞技与高价值逻辑：优先 \ci{ServerOnly}。
\end{itemize}

\textbf{一句话总结：}\\
\ci{EGameplayAbilityNetSecurityPolicy} 的四个标识符，实质是在定义“客户端对 Ability 生命周期（开始/结束）的请求权限矩阵”，从完全开放到完全服务器独占逐级收紧。

\section{AbilityTriggers 详解}

在 `GameplayAbility.h` 中，`UGameplayAbility` 通过如下成员变量保存触发配置：

\begin{code}
UPROPERTY(EditDefaultsOnly, Category = Triggers)
TArray<FAbilityTriggerData> AbilityTriggers;
\end{code}

它的作用是：\emph{定义这个 Gameplay Ability 在什么\textbf{事件或条件下}应当被自动触发（Trigger）}。  
换句话说，\ci{AbilityTriggers} 让一个 Ability 不必只能由输入直接激活，也可以响应外部游戏事件、状态变化或其他系统发送的触发信号来启动。

\subsection{核心作用}

\ci{AbilityTriggers} 本质上是一个触发规则列表。  
每个 \ci{FAbilityTriggerData} 表示一条触发配置，通常描述：

\begin{itemize}
	\item 用什么事件或信号触发；
	\item 触发时是立即激活，还是只是作为响应条件；
	\item 触发源与 Ability 之间如何关联。
\end{itemize}

因此，一个 Ability 可以同时绑定多个触发条件。例如：

\begin{itemize}
	\item 被某个 Gameplay Event 触发；
	\item 被输入映射触发；
	\item 被特定状态标签或战斗事件触发。
\end{itemize}

\subsection{在 `UGameplayAbility` 中的地位}

从 `UGameplayAbility` 的职责来看，`AbilityTriggers` 属于“\textbf{能力何时被启动}”这一层配置，和下面这些字段共同决定 Ability 是否以及如何进入激活流程：

\begin{itemize}
	\item `NetExecutionPolicy`：决定网络上在哪一端执行；
	\item `NetSecurityPolicy`：决定客户端是否有权请求执行/终止；
	\item `InstancingPolicy`：决定 Ability 如何实例化；
	\item `AbilityTriggers`：决定 Ability 何时被触发。
\end{itemize}

也就是说，`AbilityTriggers` 解决的是\textbf{触发入口}，而不是\textbf{执行位置}或\textbf{安全授权}。

\subsection{常见理解方式}

可以把 `AbilityTriggers` 理解为一个“事件订阅表”：

	\begin{longtable}{|p{0.28\linewidth}|p{0.62\linewidth}|}
		\hline
		\textbf{概念} & \textbf{解释} \\
		\hline
		触发器列表 & Ability 监听哪些事件或条件 \\
		\hline
		每条触发规则 & 某个具体事件到达时，是否允许激活该 Ability \\
		\hline
		自动触发 & 不需要玩家显式按键，也能由系统自动启动 \\
		\hline
		多触发支持 & 一个 Ability 可以响应多个不同来源的触发\\		
	\end{longtable}

\subsection{典型使用场景}

\textbf{1. 受 Gameplay Event 驱动的技能}\\
例如角色进入某种状态、命中敌人、拾取物品、完成蓄力等，系统发送事件后，Ability 自动响应。

\textbf{2. AI 或规则系统触发}\\
AI 不一定依赖玩家输入，而是根据行为树、感知系统、战斗状态等触发 Ability。

\textbf{3. 被动技能}\\
某些被动效果不是“按键施放”，而是在特定条件出现时自动生效，这类能力通常会使用触发配置。

\textbf{4. 动画或战斗流程联动}\\
例如动画通知、战斗阶段切换、连击窗口开启时触发某个 Ability。

\subsection{与输入激活的区别}

`AbilityTriggers` 与输入绑定不是同一个东西：

\begin{itemize}
	\item \textbf{输入绑定}：玩家主动按键后激活 Ability；
	\item \textbf{AbilityTriggers}：外部事件满足条件后自动激活 Ability。
\end{itemize}

前者强调“\textbf{人按下了什么}”，后者强调“\textbf{系统发生了什么}”。

\subsection{与 `ShouldAbilityRespondToEvent` 的关系}

在 `UGameplayAbility` 中，还存在：

\begin{code}
	UE_API virtual bool ShouldAbilityRespondToEvent(
	const FGameplayAbilityActorInfo* ActorInfo,
	const FGameplayEventData* Payload) const;
\end{code}

这个函数说明了 Ability 是否愿意响应某个事件。  
可以把它理解为：

\begin{itemize}
	\item `AbilityTriggers`：声明“我对哪些事件感兴趣”；
	\item `ShouldAbilityRespondToEvent`：运行时判断“这个事件我现在到底接不接”。
\end{itemize}

也就是说，\textbf{触发配置负责静态声明，响应函数负责动态裁决}。

\subsection{与 `FAbilityTriggerData` 的关系}

`AbilityTriggers` 的元素类型是 `FAbilityTriggerData`，所以真正的触发语义由该结构体决定。  
虽然你当前给出的片段里没有展开 `FAbilityTriggerData` 的定义，但从上下文可以确定：

\begin{itemize}
	\item `AbilityTriggers` 保存的是多个触发描述；
	\item 每个描述都用来告诉系统：该 Ability 应该如何被触发；
	\item 这些触发信息会被 `UGameplayAbility` 和 `AbilitySystemComponent` 用于激活判断。
\end{itemize}

\subsection{网络层面的含义}

触发器本身并不等于网络执行策略，但它会影响 Ability 在网络中的启动方式：

\begin{itemize}
	\item 如果 Ability 由客户端输入触发，则触发后还要受 `NetExecutionPolicy` 和 `NetSecurityPolicy` 约束；
	\item 如果 Ability 由服务器事件触发，则客户端不一定主动参与；
	\item 如果是可预测触发，则客户端可能先执行表现，再由服务器确认。
\end{itemize}

因此，`AbilityTriggers` 更多是“\textbf{入口条件}”，而网络策略决定“\textbf{入口之后怎么走}”。

\subsection{实践上的优缺点}

\textbf{优点：}
\begin{itemize}
	\item 让 Ability 与输入解耦；
	\item 更适合被动技能、AI 技能、事件驱动技能；
	\item 更容易做数据驱动配置；
	\item 适合设计师在编辑器中直接配置。
\end{itemize}

\textbf{缺点：}
\begin{itemize}
	\item 触发来源多时，调试会更复杂；
	\item 多触发规则可能导致重复触发或优先级冲突；
	\item 若和网络预测、取消逻辑结合不当，容易出现状态不同步。
\end{itemize}

\subsection{一句话总结}

`AbilityTriggers` 是 `UGameplayAbility` 的\textbf{触发入口配置表}：它定义了这个 Ability 会在什么事件或条件下被自动激活，是把 Ability 从“纯输入驱动”扩展为“事件驱动、系统驱动”的关键机制。

\section{Categories 元指示符}
可以在使用\ci{UPROPERTY}时，使用\ci{meta(Categories = "InputAction")}来限制\textbf{只允许}从 InputAction 这个根节点下面选。

这一节要对技能加入可以在运行期修改的\ci{Tag}，因此使用\ci{FGameplayAbilitySpec}的\ci{DynamicAbilityTags}属性。
\begin{code}
FGameplayAbilitySpec AbilitySpec(Ability, PlayerLevel, INDEX_NONE);

FGameplayTag InputTag = Ability.GetDefaultObject()->StartupInputTag;
if (InputTag.IsValid())
{
	AbilitySpec.DynamicAbilityTags.AddTag(InputTag);
}
GiveAbility(AbilitySpec);
\end{code}

相应的，在输入按键时要检查对应的\ci{Tag}
\begin{code}
for (auto& ActivatableAbilitySpec : GetActivatableAbilities())
{
	if (!ActivatableAbilitySpec.DynamicAbilityTags.HasTag(InputTag)) continue;
	if (!ActivatableAbilitySpec.IsActive())
	{
		TryActivateAbility(ActivatableAbilitySpec.Handle);
	}
}
\end{code}
\ci{FGameplayAbilitySpec}的Ability代表技能的\textbf{CDO}\sidenote{\ci{Class Default Object}}，实例根据技能的实例化策略，在调用一些最终 会调用\ci{InternalTryActivateAbility}的函数，例如\ci{TryActivateAbility}，由其根据策略生成时生成。

\part{UAbilityTask}
\section{创建AbilityTask}
在创建\ci{UAbilityTask}工厂方法时，需要类型为\ci{UGameplayAbility}的\ci{OwningAbility}变量。我们通常调用\ci{NewAbilityTask}创建Task实例\ci{MyObj}，然后\ci{MyObj->InitTask}，最终设置所属变量\ci{Ability}、所属组件\ci{AbilitySystemComponent}之类，从而可以获取所需的所有信息。

对于自定义\ci{UTargetDataUnderMouse}，需要将数据发送到服务器，以便服务端也标记目标点。可以通过GAS内置的\ci{TargetData}系统的\ci{ServerSetReplicatedTargetData}之类传递数据。


\section{向服务器发送鼠标 TargetData 的实现}


本节的核心问题是：\textbf{在 UE5 的 GAS（Gameplay Ability System）中，自定义 Ability Task 不能只在本地读取鼠标命中结果并直接广播，因为服务器端并不会自动拥有这份有效的光标命中数据。} 单机思路是：
$
\text{GetHitResultUnderCursor} \Rightarrow \text{Broadcast}
$。
但在联机 GAS 中，这样做是不够的。真正需要的是：
$
\text{客户端采集鼠标命中} \Rightarrow \text{打包为 TargetData} \Rightarrow \text{发给服务器} \Rightarrow \text{服务器接收并继续执行能力逻辑}
$

因此，这一节实际上是在补齐 GAS 技能任务中的一个关键能力链路：\textbf{客户端将目标信息作为 TargetData 通过 AbilitySystemComponent 复制到服务器。}

\subsection{问题背景：为什么“直接广播鼠标命中”不成立}

前一版实现的问题：
\begin{itemize}
	\item 当前自定义 Ability Task 是 ``TargetDataUnderMouse''。
	\item 之前只是简单地获取鼠标命中信息，然后广播出去。
	\item 这种方式在本地控制角色时看起来可用，但在网络环境下会出错。
	\item 原因是：\textbf{如果当前这一端不是 locally controlled player，那么 cursor hit 不一定有有效数据。}
\end{itemize}

这句话背后的 GAS/网络含义非常重要：

\begin{itemize}
	\item 鼠标光标是典型的\textbf{本地输入}。
	\item 本地输入天然存在于客户端，不天然存在于服务器。
	\item 服务器并不会凭空知道“玩家此刻鼠标指向了哪里”。
	\item 所以，若技能逻辑依赖鼠标位置，就必须显式把该目标信息发送给服务器。
\end{itemize}

因此，本节的根本目标不是“拿到鼠标命中”，而是：

\begin{center}
	\textbf{把本地命中的结果变成 GAS 能识别、能复制、能预测、能在服务端消费的 TargetData。}
\end{center}

\subsection{总体设计思路}

处理框架可以概括为两条分支：
\begin{enumerate}
	\item 如果当前角色是 \texttt{locally controlled}：
	\begin{itemize}
		\item 读取鼠标命中；
		\item 创建 TargetData；
		\item 打包到 TargetDataHandle；
		\item 调用 \texttt{ServerSetReplicatedTargetData} 发给服务器；
		\item 同时本地广播，保证本地预测和表现立即生效。
	\end{itemize}
	
	\item 如果当前角色不是 \texttt{locally controlled}：
	\begin{itemize}
		\item 说明当前逻辑运行在服务器或非本地代理端；
		\item 这时不应主动去读本地鼠标；
		\item 应该改为“监听客户端发送上来的 TargetData”。
	\end{itemize}
\end{enumerate}

所以本节实际是在搭建一个经典的 GAS 模式：

\[
\text{Local Input} \Rightarrow \text{Predicted Client Action} \Rightarrow \text{Replicated TargetData to Server}
\]

\subsection{Activate 中的控制分支}

字幕中首先修改的是 Ability Task 的 \texttt{Activate()} 逻辑。

核心判断是：

\begin{itemize}
	\item 通过 Ability 拿到 \texttt{CurrentActorInfo}
	\item 再通过它调用 \texttt{IsLocallyControlled()}
	\item 将结果保存为 \texttt{bIsLocallyControlled}
\end{itemize}

逻辑可以概括为：

\begin{code}
const bool bIsLocallyControlled = Ability->GetCurrentActorInfo()->IsLocallyControlled();
	
if (bIsLocallyControlled)
{
	SendMouseCursorData();
}
else
{
	// TODO: We are on the server, so listen for target data.
}
\end{code}

这里的思想非常关键：

\begin{itemize}
	\item \texttt{Activate()} 不再盲目地“读鼠标并广播”。
	\item 它先判断当前执行上下文是否具备读取本地输入的资格。
	\item 若是本地控制端，就走“发送数据”路径。
	\item 若不是，就走“等待远端发来数据”路径。
\end{itemize}

这一步让 Ability Task 从“单端取值”转变为“网络角色分工明确”的实现。

\subsection{为什么选择 TargetData 而不是直接传 Vector}

字幕随后解释了一个概念问题：\textbf{什么是 TargetData？}

它去查看了 \texttt{GameplayAbilityTargetTypes.h}，发现其中定义了多种目标数据结构，例如：

\begin{itemize}
	\item \texttt{FGameplayAbilityTargetData}
	\item \texttt{FGameplayAbilityTargetDataHandle}
	\item \texttt{FGameplayAbilityTargetData\_LocationInfo}
	\item \texttt{FGameplayAbilityTargetData\_SingleTargetHit}
\end{itemize}

本节最终选用的是：

\[
\texttt{FGameplayAbilityTargetData\_SingleTargetHit}
\]

原因非常直接：

\begin{itemize}
	\item 它专门表示“\textbf{单次命中结果}”；
	\item 内部封装的是一个 \texttt{HitResult}；
	\item 鼠标拾取本来得到的就是 \texttt{FHitResult}；
	\item 因此它与“鼠标指向哪里”这个问题天然匹配。
\end{itemize}

这一选择比直接传一个 \texttt{FVector} 更合理，主要体现在：

\begin{itemize}
	\item \texttt{FHitResult} 不仅有位置，还有法线、命中的 Actor、Component、阻挡信息等；
	\item 服务器拿到它之后拥有更完整的上下文；
	\item GAS 的很多目标处理逻辑本来就是围绕 TargetData 设计的；
	\item 这样做能与后续的能力激活、目标确认、效果应用等流程保持一致。
\end{itemize}

所以这里的设计思想是：

\begin{center}
	\textbf{不要只传最终位置，而要传 GAS 原生能理解的目标语义对象。}
\end{center}

\subsection{SendMouseCursorData 的核心实现}

这一节新增了一个私有函数：

\[
\texttt{void SendMouseCursorData();}
\]

它的工作可以拆成以下步骤。

\subsubsection{步骤一：先读取鼠标命中结果}

先通过玩家控制器执行光标射线检测，得到 \texttt{CursorHit}：

\begin{itemize}
	\item 获取 \texttt{PlayerController}
	\item 声明 \texttt{FHitResult CursorHit}
	\item 调用 \texttt{GetHitResultUnderCursor}
\end{itemize}

这是本地输入采样阶段，本质上是在客户端提取“目标”。

\subsubsection{步骤二：构造单次命中类型的 TargetData}

创建：

\begin{lstlisting}[language=C++,basicstyle=\ttfamily\small]
	FGameplayAbilityTargetData_SingleTargetHit* Data =
	new FGameplayAbilityTargetData_SingleTargetHit();
\end{lstlisting}

然后把本地得到的 \texttt{CursorHit} 填进去：

\begin{lstlisting}[language=C++,basicstyle=\ttfamily\small]
	Data->HitResult = CursorHit;
\end{lstlisting}

此时，原始的本地鼠标命中数据就被转换成了 GAS 的目标数据格式。

\subsubsection{步骤三：再创建 TargetDataHandle 进行封装}

仅有单个 \texttt{TargetData} 还不够，因为 GAS RPC 需要的是：

\[
\texttt{FGameplayAbilityTargetDataHandle}
\]

所以还要创建一个 Handle，然后把上面的 Data 放进去：

\begin{lstlisting}[language=C++,basicstyle=\ttfamily\small]
	FGameplayAbilityTargetDataHandle DataHandle;
	DataHandle.Add(Data);
\end{lstlisting}

这里可以把 \texttt{Handle} 理解为“目标数据容器”或“传输包装器”。

其意义在于：

\begin{itemize}
	\item 统一 GAS 的目标数据接口；
	\item 支持一个或多个目标数据条目；
	\item 作为 RPC 的正式参数类型；
	\item 便于后续在能力图或 C++ 中继续解包使用。
\end{itemize}

\subsection{通过 AbilitySystemComponent 把数据发给服务器}

Ability Task 内部可以方便访问 \texttt{AbilitySystemComponent}，于是本节调用了关键 RPC：

\[
\texttt{ServerSetReplicatedTargetData}
\]

这一步是全节最关键的“网络桥”。

\subsubsection{函数调用的整体意图}

这一调用的本质含义是：

\begin{center}
	\textbf{客户端告诉服务器：当前这个能力，在这个激活上下文里，我已经选好了目标，请使用这份 TargetData。}
\end{center}

\subsubsection{需要的参数}

字幕里逐一解释了该函数需要的参数。可以总结如下。

\begin{longtable}{>{\raggedright\arraybackslash}p{0.3\textwidth}
		>{\raggedright\arraybackslash}p{0.3\textwidth}
		>{\raggedright\arraybackslash}p{0.4\textwidth}}
	\toprule
	参数 & 来源 & 作用说明 \\
	\midrule
	\endhead
	
	\texttt{GameplayAbilitySpecHandle} &
	\texttt{GetAbilitySpecHandle()} &
	标识“是哪一个能力实例/规格”在发送这份 TargetData。 \\
	
	\texttt{AbilityOriginalPredictionKey} &
	\texttt{GetActivationPredictionKey()} &
	对应能力最初激活时的 PredictionKey，用来和本次能力激活上下文关联。 \\
	
	\texttt{TargetDataHandle} &
	本地构造的 \texttt{DataHandle} &
	真正要复制给服务器的目标数据内容。 \\
	
	\texttt{ApplicationTag} &
	可传空的 \texttt{FGameplayTag()} &
	附带的应用标签，本节未设置具体含义，但参数位必须满足。 \\
	
	\texttt{CurrentPredictionKey} &
	\texttt{AbilitySystemComponent ->ScopedPredictionKey} &
	当前预测上下文的 PredictionKey，用于将本次预测行为与服务端同步。 \\
	\bottomrule
\end{longtable}

于是它的典型调用形式可以概括成：

\begin{code}
AbilitySystemComponent->ServerSetReplicatedTargetData(GetAbilitySpecHandle(),
	GetActivationPredictionKey(),
	DataHandle,
	FGameplayTag(),
	AbilitySystemComponent->ScopedPredictionKey);
\end{code}

\subsubsection{这一 RPC 的工程意义}

这一调用完成了三层工作：

\begin{enumerate}
	\item \textbf{标识能力上下文}：告诉服务器是哪一次能力激活。
	\item \textbf{传递目标内容}：把命中结果送上去。
	\item \textbf{建立预测同步}：让服务器知道客户端是在预测框架中提交这份数据。
\end{enumerate}

这说明 GAS 的 TargetData 复制并不是一个“随便发个结构体”的过程，而是被严密地绑定在：

\[
\text{AbilitySpecHandle} + \text{PredictionKey} + \text{TargetDataHandle}
\]

这一整套能力网络语义之中。

\subsection{为何本地发送后还要继续广播 Delegate}

字幕特别强调：即便已经把 TargetData 发给服务器，\textbf{本地也仍然应该广播有效数据。}

原因是：

\begin{itemize}
	\item 客户端需要即时反馈；
	\item 预测式技能不应该等服务器确认后才显示本地结果；
	\item 否则玩家手感会明显变差。
\end{itemize}

因此，正确做法不是“发送给服务器以后就不管了”，而是：

\begin{center}
	\textbf{服务器同步是一条链，本地表现是另一条链，两者要并行。}
\end{center}

所以客户端在本地依然会广播 Ability Task 的 delegate，让蓝图或能力图继续向下执行。

\subsubsection{广播前必须做的安全检查}

这里字幕又补了一个重要细节：调用

\[
\texttt{ShouldBroadcastAbilityTaskDelegates()}
\]

它的作用是确认：

\begin{itemize}
	\item 当前 Ability 是否依旧处于有效、活跃状态；
	\item 若能力已经结束或失效，则不应再往 Ability Graph 回调。
\end{itemize}

因此安全模式应是：

\begin{code}
if (ShouldBroadcastAbilityTaskDelegates())
{
	ValidData.Broadcast(DataHandle);
}
\end{code}

这体现了 GAS Ability Task 的一个规范习惯：

\begin{center}
	\textbf{任何回调广播前，先检查 Ability Task 是否仍允许广播。}
\end{center}

\subsection{Delegate 参数为什么从 Vector 改成 DataHandle}

原本 delegate 可能只广播了一个 \texttt{FVector}，但字幕中明确把它改成：

\[
\texttt{const FGameplayAbilityTargetDataHandle\&}
\]

这么改的原因很充分：

\begin{itemize}
	\item 如果只传 \texttt{FVector}，就只保留了位置。
	\item 如果传 \texttt{TargetDataHandle}，就保留了完整目标语义。
	\item 后续逻辑若需要命中的 Actor、法线、碰撞信息，都还能取到。
	\item 这与 GAS 的原生输入/输出接口更一致。
\end{itemize}

因此这是一次非常重要的接口升级：

\[
\texttt{FVector} \quad \Longrightarrow \quad \texttt{FGameplayAbilityTargetDataHandle}
\]

其含义不是“类型换了”，而是：

\begin{center}
	\textbf{Ability Task 的输出从“裸坐标”提升为“可网络化、可扩展、可复用的 GAS 目标数据对象”。}
\end{center}

\subsection{PredictionKey 与预测窗口的意义}

字幕后半段引出了另一个关键话题：\textbf{PredictionKey}。

虽然讲师说明后续还会专门展开，但这一节已经透露出几个非常重要的原则。

\subsubsection{原始 PredictionKey}

\texttt{GetActivationPredictionKey()} 对应的是：

\begin{itemize}
	\item 能力最初被激活时关联的 PredictionKey；
	\item 它表明“这份 TargetData 属于哪一次能力激活预测链路”。
\end{itemize}

\subsubsection{当前 PredictionKey}

\texttt{AbilitySystemComponent->ScopedPredictionKey} 对应的是：

\begin{itemize}
	\item 当前这段预测代码执行时的 PredictionKey；
	\item 它标记“此刻这个上下文里的预测动作”。
\end{itemize}

可以粗略理解为：

\begin{itemize}
	\item \textbf{Original Key}：能力是从哪次预测激活开始的；
	\item \textbf{Current Key}：当前这个具体预测操作属于哪个预测窗口。
\end{itemize}

\subsubsection{为什么要创建 FScopedPredictionWindow}

字幕强调：\texttt{ScopedPredictionKey} 不是自动更新出来给你“随便用”的，它通常需要通过一个预测窗口来建立和管理。

因此在 \texttt{SendMouseCursorData()} 顶部创建：

\begin{lstlisting}[language=C++,basicstyle=\ttfamily\small]
	FScopedPredictionWindow ScopedPrediction(
	AbilitySystemComponent.Get()
	);
\end{lstlisting}

这一步的含义是：

\begin{itemize}
	\item 从这里开始，到当前作用域结束；
	\item 这段代码中的行为都处于一个“预测上下文”中；
	\item GAS 会生成或维护一个可用于客户端和服务器同步的 PredictionKey；
	\item 当前发送的 TargetData 将被视为“受预测保护的本地操作”。
\end{itemize}

讲师给出的理解方式可以概括为：

\begin{center}
	\textbf{我们在本地先做这件事，请服务器随后接受并对齐这件事。}
\end{center}

\subsection{FScopedPredictionWindow 的本质解释}

如果把这一工具抽象化，可以把它理解为一个 RAII 风格的预测上下文开关：

\begin{itemize}
	\item 构造时，打开预测作用域；
	\item 在作用域内，ASC 维护对应的 PredictionKey；
	\item 作用域结束时，窗口自然关闭。
\end{itemize}

它的工程价值主要有三点：

\begin{enumerate}
	\item 让本地 Ability Task 的行为可以合法地参与预测。
	\item 让服务器能通过 PredictionKey 对应到这次客户端操作。
	\item 让 GAS 的回滚、确认、同步机制具备一致的上下文依据。
\end{enumerate}

所以，这一节虽然表面上是在“发鼠标数据”，但实际上已经把 GAS 联机技能里极其关键的一层接上了：

\[
\text{TargetData Replication} + \text{Prediction}
\]

\subsection{本节的关键结论}

这一节可以归纳出以下几个必须掌握的结论：

\begin{enumerate}
	\item \textbf{鼠标目标数据属于本地输入，服务器不会自动拥有。}
	\item \textbf{在 GAS 中，应将这类目标信息封装成 TargetData，而不是只传一个坐标。}
	\item \textbf{本地控制端负责采样并通过 ASC 的 \texttt{ServerSetReplicatedTargetData()} 提交给服务器。}
	\item \textbf{TargetData 的网络提交必须与 AbilitySpecHandle 和 PredictionKey 绑定。}
	\item \textbf{发送给服务器并不意味着放弃本地广播，本地预测表现仍然需要立即执行。}
	\item \textbf{Ability Task 广播前应使用 \texttt{ShouldBroadcastAbilityTaskDelegates()} 做有效性检查。}
	\item \textbf{若涉及预测行为，应在适当作用域内创建 \texttt{FScopedPredictionWindow}。}
\end{enumerate}

\subsection{从工程实践角度的进一步拓展}

如果把本节知识放到真实项目里，可以得到几个很实用的工程结论：

\begin{itemize}
	\item 任何依赖玩家本地输入的技能目标选择，都应该优先考虑“客户端采样 + TargetData 上送”的模式。
	\item 只传位置往往不够，最好传 GAS 认可的完整目标结构。
	\item 在设计技能任务输出时，优先使用 \texttt{FGameplayAbilityTargetDataHandle}，避免后期接口反复重构。
	\item 预测窗口不是可有可无的装饰，而是 GAS 联机手感与同步一致性的关键设施。
	\item 服务端接收 TargetData 的逻辑应与本地发送逻辑成对设计，否则链路会断在中间。
\end{itemize}

\subsection{一句话总括}

本节的本质可以用一句话概括：

\begin{center}
	\textbf{它教会我们如何把“客户端鼠标点到了哪里”这件本地事实，转化为 GAS 可预测、可复制、可由服务器权威处理的 TargetData。}
\end{center}

\section{PredictionKey 在 GAS 中的同步机制}

虚幻引擎的 Gameplay Ability System（GAS）通过 \texttt{FPredictionKey} 实现\textbf{客户端预测}（Client Prediction）与\textbf{服务器权威}（Server Authority）之间的状态同步。其核心思想是：客户端先本地执行技能（预测），同时将预测标识发送给服务端；服务端执行后用相同标识确认或纠正，客户端据此决定是保留预测结果还是回滚。

\subsection{PredictionKey 的结构与生成}

\texttt{FPredictionKey} 本质上是一个递增的序列号，主要包含两个字段：

\begin{code}
struct FPredictionKey
{
	int32 Current;      // 当前预测序号
	int32 Base;         // 基准序号（用于回滚定位）
	UObject* Predictor; // 预测发起者（通常是 PlayerController）
};
\end{code}

每次客户端发起预测性操作时，序号自增，确保每一次预测都有唯一标识。

\subsection{预测执行的完整流程}

\subsubsection{1. 客户端发起预测}

当客户端按下技能键时：

\begin{enumerate}
	\item 客户端生成新的 \texttt{FPredictionKey}（序号 +1）
	\item 本地立即执行 Ability（播放动画、应用 GameplayEffect 等），称为 \textbf{Predictive Ability}
	\item 将 \texttt{PredictionKey} 连同技能激活请求一起通过 RPC 发送给服务端
	\item \textbf{所有}由该 Ability 触发的 Task、Effect 都携带此 Key，形成一条预测链
\end{enumerate}

\subsubsection{2. 服务端验证与执行}

服务端收到请求后：

\begin{enumerate}
	\item 检查技能是否合法（冷却、消耗、是否被打断等）
	\item 使用客户端传来的同一个 \texttt{PredictionKey} 在服务端执行 Ability
	\item 执行结果通过 \texttt{Replicate} 同步回客户端
	\item 如果服务端执行失败或结果不同，会触发 \textbf{回滚（Rollback）}
\end{enumerate}

\subsubsection{3. 客户端确认或回滚}

客户端收到服务端同步后：

\begin{itemize}
	\item \textbf{成功确认}：服务端结果与预测一致 → 什么都不做，预测结果保留
	\item \textbf{失败回滚}：服务端拒绝执行 → 客户端撤销该 PredictionKey 对应的所有预测效果（移除 GameplayEffect、恢复属性等）
	\item \textbf{差异纠正}：服务端执行成功但数值不同 → 客户端以服务端为准覆盖预测值
\end{itemize}

\subsection{AbilityTask 中的 PredictionKey 用法}

在 \texttt{UAbilityTask} 中，\texttt{GetActivationPredictionKey()} 返回激活当前 Ability 时使用的预测 Key：

\begin{code}
	// 在 Task 中判断是否处于预测上下文
	if (AbilitySystemComponent->IsPredictingClient())
	{
		FPredictionKey Key = GetActivationPredictionKey();
		// 用 Key 标记预测性操作
	}
\end{code}

常见的使用场景包括：

\begin{itemize}
	\item \textbf{WaitTargetData}：客户端预测生成目标数据，服务端验证后确认
	\item \textbf{ApplyGameplayEffect}：预测性应用效果，服务端确认后正式生效
	\item \textbf{PlayMontage}：客户端先播动画，服务端确认后同步给其他客户端
\end{itemize}

\subsection{回滚机制的实现原理}

GAS 的回滚基于 \textbf{预测链（Prediction Chain）}：

\begin{enumerate}
	\item 每个预测性操作都记录在 \texttt{FPredictionKey} 对应的链表中
	\item 属性修改（AttributeSet）会保存修改前的值，挂在对应 Key 下
	\item GameplayEffect 的预测应用会标记为 \texttt{Predicted}，而非真正永久生效
	\item 当服务端确认某 Key 成功时，该 Key 之前的所有预测都被"提交"
	\item 当服务端拒绝某 Key 时，从该 Key 开始往后的所有预测效果全部撤销
\end{enumerate}

这就是为什么 \texttt{FPredictionKey} 有 \texttt{Base} 字段——它标记了上一个已确认的预测点，回滚时只需回退到 \texttt{Base} 对应的状态即可。

\subsection{关键设计要点}

\begin{itemize}
	\item \textbf{服务端权威}：最终状态始终以服务端为准，预测只是为了手感
	\item \textbf{增量确认}：服务端不必每个 Key 都回复，只需定期确认最新成功的 Key
	\item \textbf{链式依赖}：如果 Key=5 被回滚，Key=6、7、8 等后续预测也全部作废
	\item \textbf{网络优化}：预测成功时不产生额外流量，失败时才发送纠正数据
\end{itemize}

\section{FScopedPredictionWindow 的作用与原理}

\texttt{FScopedPredictionWindow} 是 GAS 中一个 \textbf{RAII 风格的作用域辅助类}，用于在一段代码执行期间\textbf{临时设置预测上下文}，执行完毕后自动恢复。

\subsection{核心作用}

它的本质是一个\textbf{预测 Key 的``作用域管理器''}：

\begin{itemize}
	\item \textbf{构造时}：将指定的 \texttt{FPredictionKey} 设置为当前激活的预测上下文
	\item \textbf{析构时}：自动恢复之前的预测上下文
\end{itemize}

这样可以确保作用域内的所有操作（如应用 GameplayEffect、修改属性）都被正确标记为“预测性操作”，出了作用域自动恢复，不会影响后续代码。

\subsection{典型用法}

\begin{code}
	// 在 AbilityTask 或 Ability 中
	void UMyAbilityTask::OnSomethingHappened()
	{
		// 开启预测窗口
		FScopedPredictionWindow ScopedPrediction(AbilitySystemComponent, GetActivationPredictionKey());
		
		// 这个作用域内的所有操作都会被标记为预测
		ApplyGameplayEffect();   // 预测性应用效果
		ModifyAttribute();       // 预测性修改属性
		SpawnActor();            // 预测性生成Actor
		
	} // 离开作用域，自动恢复之前的预测状态
\end{code}

\subsection{为什么需要它}

\subsubsection{1. 预测链的正确追踪}

GAS 的回滚机制依赖于**\textbf{每个预测操作都关联到正确的 PredictionKey}**。如果操作发生时当前预测上下文不对，就会导致：

\begin{itemize}
	\item 回滚时漏掉某些操作 $\rightarrow$ 状态不一致
	\item 操作被错误地归到其他预测 Key 下 $\rightarrow$ 回滚范围错误
\end{itemize}

\texttt{FScopedPredictionWindow} 保证了作用域内的操作都归属到指定 Key。

\subsubsection{2. 嵌套预测的安全处理}

有时候预测操作会嵌套调用（比如一个 Task 触发另一个 Task），如果直接设置全局预测 Key，内层返回后外层的上下文就丢了。

\texttt{FScopedPredictionWindow} 会**\textbf{保存并恢复}**之前的状态，支持安全嵌套：

\begin{code}
	void Outer()
	{
		FScopedPredictionWindow Outer(KeyA);
		// 当前 Key = KeyA
		
		Inner(); // 内部临时切换到 KeyB
		
		// 回到这里，Key 自动恢复为 KeyA
	}
	
	void Inner()
	{
		FScopedPredictionWindow Inner(KeyB);
		// 当前 Key = KeyB
	}
\end{code}

\subsubsection{3. 服务端与客户端的统一逻辑}

同一个函数在客户端和服务端都会执行：

\begin{itemize}
	\item \textbf{客户端}：\texttt{ScopedPrediction} 设置预测 Key，操作被标记为预测
	\item \textbf{服务端}：同样的代码，但因为是权威端，预测 Key 实际不生效（或用于确认）
\end{itemize}

这样同一份代码在两端都能正确运行，不需要写两套逻辑。

\subsection{内部实现原理}

\texttt{FScopedPredictionWindow} 内部很简单，大概做了这几件事：

\begin{enumerate}
	\item 保存旧的 \texttt{PredictionKey} 和预测模式
	\item 调用 \texttt{AbilitySystemComponent->SetCurrentPredictionKey(NewKey, bIsPredicting)}
	\item 析构时恢复保存的旧值
\end{enumerate}

它还会处理一些细节：

\begin{itemize}
	\item 只在客户端预测模式下才真正切换（服务端跳过）
	\item 处理 \texttt{Predictor} 对象的有效性检查
	\item 处理网络相关性（只有本地控制的玩家才预测）
\end{itemize}

\subsection{与 AbilityTask 的关系}

在 \texttt{UAbilityTask} 中，很多内置 Task（如 \texttt{WaitTargetData}、\texttt{ApplyGameplayEffect}）内部都会自动使用 \texttt{FScopedPredictionWindow}，所以写普通 Task 时通常不需要手动写。

但在以下情况需要手动创建：

\begin{itemize}
	\item \textbf{自定义 Task} 中执行预测性操作
	\item \textbf{异步回调}中（如定时器、动画事件），因为回调执行时预测上下文可能已经丢失
	\item \textbf{跨 Task 调用}时，需要显式传递预测 Key
\end{itemize}

\subsection{总结}

\texttt{FScopedPredictionWindow} 就是预测 Key 的**"\textbf{临时切换器}"**——用 RAII 方式保证一段代码在正确的预测上下文中执行，用完自动恢复，是 GAS 预测机制能安全、正确工作的基础设施之一。
